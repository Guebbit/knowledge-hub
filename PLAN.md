# 2repo — Implementation Plan

New sibling command to `2brain` focused on understanding and documenting codebases.
Two outputs: a **knowledge graph** that lives in the target repo (saves Claude Code tokens)
and an **llm-wiki** that lives in the Obsidian vault (human reference).

---

## Decision checklist

Mark each `[ ]` as `[x]` when decided, then hand this back to implement.

---

## 1. New command: `2repo`

**Agreed:** Option B — a new sibling script `scripts/2repo.sh`, mirroring `2brain.sh`.
Registered as a global alias in `~/.zshrc`:

```bash
alias 2repo="$KNOWLEDGE_HUB/scripts/2repo.sh"
```

Internally calls a new Python entry point `scripts/repo.py` (inside the scripts container,
same Docker setup as `main.py`).

---

## 2. graphify — knowledge graph in the repo

A single generated file that lives **inside the target repo**, not in the vault.
Purpose: let Claude Code load it at session start instead of reading dozens of files.

### 2a. Output location

- [x] **`knowledge-graph.md` at repo root** — visible, agnostic (not tied to Claude Code), human + AI readable
- A `.claude/KNOWLEDGE.md` (or similar) is written alongside it containing just `@../knowledge-graph.md` so Claude Code auto-loads it, without the graph itself being hidden

> Decided: root-level graph for visibility/agnosticism, `.claude/` reference file for Claude Code auto-load.

### 2b. CLAUDE.md integration

When graphify runs, it can also write or update the repo's `CLAUDE.md` to reference the graph:

```markdown
@.claude/knowledge-graph.md
```

This makes the graph load automatically every Claude Code session, zero flags needed.

- [x] **Always inject the `@` reference into `CLAUDE.md`** (creates it if missing, appends if present)
- [ ] **Only inject if `--inject-claude-md` flag is passed**
- [ ] **Never touch `CLAUDE.md`, user adds the reference manually**

If injecting: use clear markers so it can be updated without clobbering manual content:

```markdown
<!-- 2repo:start — regenerate with: 2repo ~/Work/my-repo --graph -->
@.claude/knowledge-graph.md
<!-- 2repo:end -->
```

### 2c. Commit vs gitignore

- [x] **Commit it** — shared with the team, everyone benefits from the graph
- [ ] **Gitignore it** — personal context layer, not part of the codebase
- [ ] **`2repo --install-hook` adds it to `.gitignore` automatically** — opt-in per repo

> If committed: the graph becomes documentation. If gitignored: it's a private AI assistant file.

### 2d. Graph file format

The file is markdown so Claude Code can read it directly. Contents:

```markdown
<!-- 2repo:generated | date: 2026-06-21 | files: 47 | hash: a3f8c2 -->
# Knowledge Graph — my-repo

## Purpose
One-paragraph description of what this repo does.

## Module map
\`\`\`mermaid
graph TD
  src/index.ts --> src/routes/auth.ts
  src/routes/auth.ts --> src/controllers/AuthController.ts
  src/controllers/AuthController.ts --> src/services/UserService.ts
  src/services/UserService.ts --> src/db/queries/users.ts
\`\`\`

## Key files
| File | Purpose |
|---|---|
| `src/index.ts` | Entry point, mounts all routers |
| `src/db/connection.ts` | Singleton DB pool, imported everywhere |
| `src/config.ts` | Env var parsing, imported at startup |

## Conventions
- All DB calls go through `src/db/queries/`, never raw SQL elsewhere
- All async functions return `Result<T, E>`, never throw
- Tests in `__tests__/` mirror the source tree

## What to avoid
- Do not edit `src/generated/` — auto-generated by `npm run codegen`
- Do not add business logic to route handlers, use services
```

- [x] **Include Mermaid diagram** (visual in Obsidian, parseable by Claude)
- [ ] **Skip Mermaid, just structured text** (faster generation, always readable)
- [ ] **Both: Mermaid for medium/deep, text-only for shallow**

## 3. llm-wiki — Obsidian vault + repo pointer

### 3a. Vault structure

Wiki lands in `vault/Projects/<repo-name>/`:

```
vault/Projects/my-repo/
├── index.md          ← hub note: overview, links to all pages below
├── graph.md          ← Mermaid diagram (richer version of the repo graph)
├── architecture.md   ← layers, key decisions, why things are structured this way
├── setup.md          ← how to run, dev, configure, deploy
└── components/       ← one note per key module (deep only)
    ├── auth.md
    └── database.md
```

Pages generated per depth:

| Depth    | Notes created                                      |
|----------|----------------------------------------------------|
| shallow  | `index.md` only                                    |
| medium   | `index.md` + `graph.md` + `architecture.md` + `setup.md` |
| deep     | all above + one note per key module (AI decides how many) |

All notes are cross-linked via `[[wikilinks]]` so the Obsidian graph view shows the repo as a connected cluster.

### 3b. Repo pointer

A small file in the target repo pointing back to the vault wiki:

```
my-repo/.claude/wiki.md
```

Contents:
```markdown
# Wiki
Obsidian vault: ~/knowledge-hub/vault/Projects/my-repo/index.md
Generated: 2026-06-21 with `2repo ~/Work/my-repo --wiki`
To regenerate: `2repo ~/Work/my-repo --wiki`
```

- [x] **Always create the pointer file**. `--no-wiki` to ignore
- [ ] **Only create the pointer if `--wiki` was used** (no pointer for graph-only runs)

### 3c. llm-wiki necessity

- [x] **Both graph + wiki always** (default, one command does everything). `--no-graph` and `--no-wiki` to remove one or the oteher or both
- [ ] **Separate flags: `--graph` and `--wiki`** (user opts into each)
- [ ] **`2repo` defaults to graph only, `--wiki` adds the vault output** (graph is the token-saver, wiki is extra)

> Recommendation: separate flags, graph as default since that's the primary token-saving use case.

## 4. Depth levels

Controls how much of the target repo is read before calling the AI.

### What each depth reads

| Input                                    | shallow | medium | deep |
|------------------------------------------|---------|--------|------|
| File tree (`find` output)                | yes     | yes    | yes  |
| `README.md`                              | yes     | yes    | yes  |
| Package manifest (`package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, etc.) | yes | yes | yes |
| `.env-example` / `config.*`              | no      | yes    | yes  |
| Main entry points (`main.py`, `index.ts`, `app.*`, `cmd/`) | no | yes | yes |
| `docker-compose.yml`, `Dockerfile`       | no      | yes    | yes  |
| All source files (chunked if large)      | no      | no     | yes  |
| Test files                               | no      | no     | yes  |
| **Approx. tokens fed to AI**             | ~3k     | ~15k   | ~80k+ |

### Default depth

- [ ] **shallow** — fast, cheap, good for quick overviews
- [ ] **medium** — balanced, recommended default
- [ ] **deep** — thorough, requires large context model
- [x] **hybrid** — deep by default, but if it fails notify the user then proceed with medium

## 5. Preset wiring

Depth and preset are orthogonal: depth controls input volume, preset controls which AI handles it.
They can be linked in `.env` so the right model is used automatically per depth.

### New `.env` variables (add to `.env-example`)

```bash
# ── 2REPO ─────────────────────────────────────────────────────────────────────
# Which preset to use per depth (references existing PRESET_* names)
REPO_PRESET_SHALLOW=local        # uses PRESET_LOCAL (fast Ollama model)
REPO_PRESET_MEDIUM=local         # same model, more input
REPO_PRESET_DEEP=smart           # uses PRESET_SMART (Claude/GPT-4, large context)
```

### CLI override always available

```bash
2repo ~/Work/my-repo --deep --preset cheap   # override preset for this run
```

### Open decisions

- [x] **Use separate `REPO_PRESET_*` variables** (as above)
- [ ] **Reuse `DEFAULT_PRESET` for all depths, no repo-specific preset config**
- [ ] **Add `REPO_DEFAULT_PRESET` as a single override (same model for all depths)**

## 6. Staleness detection

### Layer 1 — embedded metadata (always)

The graph file stores a fingerprint at generation time:

```
<!-- 2repo:generated | date: 2026-06-21 | files: 47 | hash: a3f8c2 -->
```

Hash is computed from: sorted file tree + modification times of key files. Cheap, no AI call.

### Layer 2 — `--check` command

```bash
2repo --check ~/Work/my-repo
```

Recomputes the hash, compares to embedded value, reports:
- up to date
- stale: N files changed, last generated DATE
- no graph found

No AI call, instant.

### Layer 3 — git hook (opt-in)

```bash
2repo --install-hook ~/Work/my-repo
```

Drops a `post-commit` hook in `<repo>/.git/hooks/post-commit` that runs `2repo --check` and prints a warning if stale. Does **not** auto-regenerate (avoids slow/expensive AI calls blocking git).

```
[2repo] knowledge-graph.md may be stale (12 files changed since last generation)
        Run: 2repo ~/Work/my-repo --graph
```

- [x] **Include all three layers**
- [ ] **Skip git hook layer (just --check manually)**
- [ ] **Auto-regenerate on post-commit if stale** (slower, burns API credits on every commit)

### What counts as "changed enough to regenerate"

- [ ] **Any file change** (most sensitive, most noisy)
- [ ] **Source files only** (ignore docs, assets, lock files)
- [x] **>N files changed** (configurable threshold, e.g. `REPO_STALE_THRESHOLD=5`)
- [ ] **Key files only** (entry points + manifest; structural changes only)

## 7. Full CLI reference

```bash
# Generate graph + wiki (default depth from REPO_DEFAULT_DEPTH)
2repo ~/Work/my-repo

# Outputs
2repo ~/Work/my-repo --graph          # graph only (in repo)
2repo ~/Work/my-repo --wiki           # wiki only (in vault)
2repo ~/Work/my-repo --graph --wiki   # both (explicit)

# Depth
2repo ~/Work/my-repo --shallow
2repo ~/Work/my-repo --medium
2repo ~/Work/my-repo --deep

# Preset override
2repo ~/Work/my-repo --preset smart

# Staleness
2repo --check ~/Work/my-repo
2repo --install-hook ~/Work/my-repo

# Target folder in vault (default: Projects)
2repo ~/Work/my-repo --wiki -f Reference
```

## 9. Implementation breakdown

Files to create:

| File | What |
|---|---|
| `scripts/2repo.sh` | Shell wrapper, mirrors `2brain.sh` |
| `scripts/repo.py` | Main entry point for `2repo` |
| `scripts/repo_scanner.py` | Reads target repo at requested depth, returns structured context |
| `scripts/repo_graph.py` | Calls AI, generates graph markdown |
| `scripts/repo_wiki.py` | Calls AI, generates wiki notes |
| `scripts/repo_staleness.py` | Hash computation + `--check` logic |

Files to modify:

| File | What |
|---|---|
| `.env-example` | Add `REPO_*` variables |
| `docker-compose.yml` | Mount target repos into container (needs a strategy — see below) |
| `README.md` | Add `2repo` section |

### Container mount challenge

Same pattern as `--from-file` in `2brain.sh`: the shell wrapper adds a volume at runtime.

```bash
-v "$(realpath $REPO_PATH):/target-repo:rw"
```

`:rw` because the container must both read the repo (scanning) and write back `knowledge-graph.md`.
The vault is already mounted in `docker-compose.yml`, so wiki output works without any extra volume.

- [x] **Per-call volume mount** — `2repo.sh` mirrors the `--from-file` pattern already proven in `2brain.sh`
